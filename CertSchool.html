01 — Understand
How Certificates Actually Work

Before generating anything, understand the core concepts. Each card below explains a fundamental building block of the PKI (Public Key Infrastructure) system that secures the internet.

🔑
Public/Private Key Pairs
The foundation of all certificate operations

Every certificate starts with a key pair: a private key (kept secret) and a public key (shared freely). Data encrypted with one can only be decrypted with the other.

When you generate a certificate here, we create an RSA key pair. The server.key file is your private key — never share it. The public key gets embedded inside the certificate itself.

📜
X.509 Certificates
The digital identity card for servers

An X.509 certificate is a standardized document that binds a public key to an identity (like a domain name). It's what your browser checks when you see the 🔒 padlock.

It contains: the subject (who it identifies), the issuer (who vouches for it), validity dates, the public key, and extensions like SAN entries. The whole thing is digitally signed.

🏛️
Certificate Authority (CA)
The trusted entity that signs certificates

A CA is an entity whose job is to verify identity and sign certificates. Your browser ships with ~150 pre-trusted root CAs (like Let's Encrypt, DigiCert).

For development, you can create your own CA — that's step 1 below. Import its ca.crt into your OS trust store once, and every certificate it signs will be trusted automatically.

🌐
SAN — Subject Alternative Name
Why CN alone doesn't work anymore

Since 2017, browsers ignore the Common Name (CN) field for hostname verification. Instead, they check the SAN extension for the list of valid DNS names and IPs.

SAN entries like DNS.1 = myapp.local and IP.1 = 127.0.0.1 tell the browser exactly which hostnames this certificate is valid for. No SAN = browser rejection.

✍️
Self-Signed vs. CA-Signed
Two paths, different trust models

Self-signed: the certificate signs itself with its own key. Quick and easy, but every device must explicitly trust this specific certificate.

CA-signed: a separate CA key signs the certificate. Any device that trusts the CA automatically trusts everything it signs — much easier to manage at scale. This is how the real internet works.

📋
CSR — Certificate Signing Request
How you ask a CA to sign your certificate

A CSR is a formal request containing your public key + identity info, signed by your private key. You send this to a CA, and they send back a signed certificate.

The CSR proves you own the private key without revealing it. In the CA-signed flow (step 2b), the tool generates the CSR automatically and signs it with your local CA.

✅ CA Ready! How to Trust Your New Certificates:

To make your browsers trust the certificates you generate, you must import ca.crt into your operating system's trust store:

  • Windows: Double-click ca.crt → Install Certificate → Local Machine → Place all certificates in the following store: "Trusted Root Certification Authorities".
  • macOS: Double-click ca.crt to open Keychain Access. Find the imported certificate in the System keychain, double-click it, expand the "Trust" section, and set "When using this certificate" to "Always Trust".
  • Linux: Copy ca.crt to /usr/local/share/ca-certificates/, and run sudo update-ca-certificates.
  • Firefox: Firefox uses its own store. Go to Settings → Privacy & Security → View Certificates → Authorities → Import, select ca.crt, and check "Trust this CA to identify websites".

Note: Never import a CA unless you generated it yourself or completely trust the source. Once imported, restart your browser.
